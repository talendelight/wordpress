## 1. Overview

### 1.1 Purpose
Provide Managers with a centralized hub for system administration tasks. This page serves as the entry point for various admin operations including user registration request approvals, system configuration, user management, and other administrative functions that require manager-level access.

### 1.2 User Role
**Manager:** Internal employee with administrative privileges to approve/reject user data change requests.

### 1.3 Navigation Path
`Managers Dashboard → Admin Tile`

### 1.4 URL Structure
- **Main Page:** `/managers/admin/`
- **Page Title:** Manager Admin
- **Hero Heading:** Admin Operations

### 1.5 Access Control
- **Allowed Roles:** Manager, Administrator
- **Denied Roles:** All others (show 403 Forbidden)
- **Plugin:** Uses existing `talendelight-roles` plugin for RBAC

## 2. Database Schema

### 2.1 Existing Table
**Table:** `wp_td_user_data_change_requests`

**Current Schema (from v3.3.0):**
```sql
CREATE TABLE wp_td_user_data_change_requests (
    id BIGINT(20) UNSIGNED NOT NULL AUTO_INCREMENT,
    user_id BIGINT(20) UNSIGNED NOT NULL,
    request_type VARCHAR(50) NOT NULL,
    field_name VARCHAR(100) NOT NULL,
    old_value TEXT,
    new_value TEXT NOT NULL,
    status VARCHAR(20) NOT NULL DEFAULT 'pending',
    requested_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    reviewed_at DATETIME,
    reviewed_by BIGINT(20) UNSIGNED,
    review_notes TEXT,
    PRIMARY KEY (id),
    KEY user_id (user_id),
    KEY status (status),
    KEY requested_at (requested_at)
);
```

**Status Values:**
- `pending` - Awaiting manager review
- `approved` - Approved and applied
- `rejected` - Rejected with reason
- `expired` - Auto-expired after 30 days

**Request Types:**
- `profile_update` - General profile information changes
- `contact_change` - Email, phone number updates
- `role_change` - User requesting different role (rare, requires verification)
- `account_deletion` - User requesting account deletion (GDPR compliance)
- `data_correction` - Correction of incorrect information

### 2.2 Related Tables
- `wp_users` - User accounts (user_id reference)
- `wp_usermeta` - User metadata (where changes are applied)

### 4.3 Action Buttons

**Approve & Apply Changes:**
- ✅ Immediately updates user data
- Sets status to `approved`
- Records review timestamp and manager ID
- Sends confirmation email to user
- Logs action in audit trail

**Reject Request:**
- ❌ Sets status to `rejected`
- Requires review notes (mandatory)
- Sends rejection email to user with reason
- Does NOT apply changes
- Logs action in audit trail

**Flag for Further Review:**
- ⏸ Marks request as needing additional attention
- Sends to separate queue (optional feature)
- Can assign to another manager
- User notified of delay

**Delete Request (Rare):**
- 🗑 Permanently removes request
- Requires confirmation modal
- Only for spam/invalid requests
- Logs deletion in audit trail

### 7.2 State Transitions

**Pending → Approved:**
- Manager clicks "Approve"
- Changes applied to user profile
- User notified via email
- Status updated in database
- Audit log created

**Pending → Rejected:**
- Manager clicks "Reject" with reason
- No changes applied
- User notified with rejection reason
- Status updated in database
- Audit log created

**Pending → Expired:**
- Auto-triggered after 30 days
- Cron job runs daily at 2 AM
- User notified of expiration
- Can resubmit request

**Approved/Rejected → Archived:**
- After 90 days, moved to archive table
- Accessible via "View History" link
- Not shown in main table

## 8. Validation Rules

### 8.1 Auto-Validation (Pre-Manager Review)

**Phone Number:**
- ✅ Must be valid E.164 format
- ✅ Must not be duplicate (if unique constraint)
- ✅ Must include country code
- ⚠️ Warn if differs significantly from current (e.g., country change)

**Email Address:**
- ✅ Must be valid email format
- ✅ Must not exist in system (unique constraint)
- ✅ Must pass DNS validation (MX record exists)
- ⚠️ Warn if free email provider (gmail, yahoo) for Employer accounts

**Profile Fields:**
- ✅ Must not contain profanity/inappropriate content
- ✅ Must meet length requirements
- ✅ Must pass XSS/injection checks

**Account Deletion:**
- ⚠️ Always flagged as Critical priority
- ⚠️ Check for outstanding obligations (unpaid invoices, active contracts)
- ⚠️ Warn if account < 30 days old (potential fraud)

### 8.2 Risk Scoring

**Low Risk (Auto-Approve Candidate):**
- First-time change
- Account > 30 days old
- Standard profile field (bio, job title)
- No red flags in validation

**Medium Risk (Manager Review):**
- Second change in 30 days
- Contact information (phone, email)
- Account < 30 days old
- Minor validation warnings

**High Risk (Manager + Flag):**
- Multiple changes in short period
- Email change for high-value employer
- Account deletion request
- Failed validation checks
- Suspicious patterns (bot-like behavior)

**Future Enhancement:** Machine learning risk scoring based on historical patterns.

## 9. Compliance & Audit

### 9.1 GDPR Compliance

**Right to Access:**
- Users can view all their pending/historical requests
- Available in "My Account" section
- Shows status and review notes (if applicable)

- Account deletion requests handled here

### 10.3 Database Queries

**Get Pending Requests (with filters):**
```php
$wpdb->prepare("
    SELECT 
        r.*,
        u.display_name,
        u.user_email,
        m.meta_value as user_role
    FROM {$wpdb->prefix}td_user_data_change_requests r
    INNER JOIN {$wpdb->users} u ON r.user_id = u.ID
    INNER JOIN {$wpdb->usermeta} m ON u.ID = m.user_id AND m.meta_key = 'wp_capabilities'
    WHERE r.status = %s
    AND r.request_type = %s
    AND r.requested_at >= %s
    ORDER BY 
        CASE r.priority 
            WHEN 'critical' THEN 1
            WHEN 'high' THEN 2
            ELSE 3
        END ASC,
        r.requested_at ASC
    LIMIT %d OFFSET %d
", $status, $type, $date_from, $limit, $offset);
```

**Approve Request:**
```php
// 1. Start transaction
$wpdb->query('START TRANSACTION');

// 2. Update user data
$wpdb->update(
    $wpdb->usermeta,
    ['meta_value' => $new_value],
    ['user_id' => $user_id, 'meta_key' => $field_name]
);

// 3. Update request status
$wpdb->update(
    "{$wpdb->prefix}td_user_data_change_requests",
    [
        'status' => 'approved',
        'reviewed_at' => current_time('mysql'),
        'reviewed_by' => get_current_user_id(),
        'review_notes' => $notes
    ],
    ['id' => $request_id]
);

// 4. Create audit log
$wpdb->insert(
    "{$wpdb->prefix}td_audit_log",
    [
        'event_type' => 'request_approved',
        'entity_type' => 'user_data_change_request',
        'entity_id' => $request_id,
        'user_id' => $user_id,
        'manager_id' => get_current_user_id(),
        'action' => 'approve',
        'old_value' => $old_value,
        'new_value' => $new_value,
        'ip_address' => $_SERVER['REMOTE_ADDR'],
        'user_agent' => $_SERVER['HTTP_USER_AGENT']
    ]
);

// 5. Commit transaction
$wpdb->query('COMMIT');

// 6. Send email notification
wp_mail($user_email, 'Request Approved', $message);
```

// Register cron hook
add_action('td_expire_old_requests', 'td_expire_old_user_requests');

function td_expire_old_user_requests() {
    global $wpdb;
    
    $expired_date = date('Y-m-d H:i:s', strtotime('-30 days'));
    
    $wpdb->query($wpdb->prepare("
        UPDATE {$wpdb->prefix}td_user_data_change_requests
        SET status = 'expired'
        WHERE status = 'pending'
        AND requested_at < %s
    ", $expired_date));
    
    // Send expiration emails to users

}

// Schedule cron (on plugin activation)

    wp_schedule_event(strtotime('02:00:00'), 'daily', 'td_expire_old_requests');
}
