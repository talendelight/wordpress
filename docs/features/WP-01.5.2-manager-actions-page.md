## Architecture

### Page Structure
```
/managers/actions
├── Hero Section (Navy background)
├── 5-Tab Interface
│   ├── Tab 1: New (Unassigned requests)
│   ├── Tab 2: Assigned (My assigned requests)
│   ├── Tab 3: Approved (90-day history)
│   ├── Tab 4: Rejected (90-day history)
│   └── Tab 5: All (90-day history, all statuses)
└── Footer (GDPR compliance badges)
```

### Tab Definitions

**Tab 1: New** (Status: "new")
- **Purpose:** Unassigned registration requests awaiting initial triage
- **Actions:** ➜ Assign (32px navy circle)
- **Limit:** 50 most recent
- **Target Users:** All public/internal registration requests not yet assigned

**Tab 2: Assigned** (Status: "assigned")
- **Purpose:** Requests assigned to current Manager for review
- **Actions:** ✓ Approve (green), ✗ Reject (red)
- **Limit:** 50 most recent
- **Filter:** Assigned to current user only

**Tab 3: Approved** (Status: "approved")
- **Purpose:** Recently approved requests (audit trail)
- **Actions:** ↶ Undo Approve (orange) - reverts to "assigned"
- **Limit:** 50 most recent
- **Date Range:** Last 90 days

**Tab 4: Rejected** (Status: "rejected")
- **Purpose:** Recently rejected requests (audit trail)
- **Actions:** ↶ Undo Reject (orange) - reverts to "assigned"
- **Limit:** 50 most recent
- **Date Range:** Last 90 days

**Tab 5: All** (Status: "all")
- **Purpose:** Complete history for reporting and auditing
- **Actions:** None (read-only)
- **Limit:** 100 most recent
- **Date Range:** Last 90 days

## Technical Implementation

### MU-Plugin: `manager-actions-display.php` (v1.0.0, 557 lines)

**Shortcode:** `[manager_actions_table status="..." limit="..." days="..."]`

**Parameters:**
- `status` (required): "new", "assigned", "approved", "rejected", "all"
- `limit` (optional): Max records to display (default: 50)
- `days` (optional): History window in days (default: 90)

**Functions:**
1. `td_manager_actions_table_shortcode()` - Shortcode handler
2. `td_manager_actions_enqueue_scripts()` - Load CSS/JS
3. `td_manager_actions_handle_ajax()` - AJAX action processor
4. Role check: Blocks non-Manager/Administrator users (returns error message)

**Database Integration:**
- Table: `td_user_data_change_requests`
- Columns: id, email, first_name, last_name, role_requested, status, assigned_to, created_at, updated_at
- Query: Filtered by status, assigned_to (for "assigned" tab), date range

**AJAX Actions:**
- `td_manager_action_assign` - Assign request to Manager
- `td_manager_action_approve` - Approve request (status → "approved", generate record_id)
- `td_manager_action_reject` - Reject request (status → "rejected")
- `td_manager_action_undo_approve` - Revert approved → assigned
- `td_manager_action_undo_reject` - Revert rejected → assigned

**Action Buttons (32px circles):**
- **➜ Assign** (Navy #063970) - Opens modal with Manager dropdown
- **✓ Approve** (Green #4CAF50) - Inline confirmation, immediate execution
- **✗ Reject** (Red #F44336) - Opens modal for rejection reason (required)
- **↶ Undo** (Orange #FF9800) - Opens confirmation modal

**Security:**
- Nonce verification on all AJAX calls
- Role check (Manager/Administrator only)
- No self-approval enforcement (future enhancement)
- SQL injection prevention (prepared statements)

## User Workflow

### Typical Manager Workflow

1. **Initial Triage (Tab 1: New)**
   - Review unassigned requests
   - Click ➜ button → Assign modal opens
   - Select Manager (self or colleague)
   - Request moves to "Assigned" status

2. **Review & Decision (Tab 2: Assigned)**
   - View assigned requests
   - Review details (email, name, role)
   - Click ✓ to approve → Status = "approved", Record ID generated
   - Click ✗ to reject → Modal opens, enter reason → Status = "rejected"

3. **Audit & Corrections (Tabs 3-4)**
   - Review recently approved/rejected requests
   - Click ↶ to undo → Status reverts to "assigned"
   - Useful for accidental approvals/rejections

4. **Reporting (Tab 5: All)**
   - View complete history (90 days)
   - No actions available (read-only)
   - Export functionality (future)
