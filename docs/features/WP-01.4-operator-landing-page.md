## 1. Overview

### 1.1 Purpose
Internal portal page for Operators to manage their daily work: candidates, employers, scouts, and operational tasks. The page provides an action-oriented dashboard with drill-down navigation to detailed views.

### 1.2 User Role
**Operator:** Internal employee handling candidate intake, employer requests, and scout management.

### 1.3 Role Capabilities
Operator has ALL Scout capabilities plus:
- Manage candidates (view, assign, update status, add notes)
- Manage employers (view requests, update status, communicate)
- Track scouts (view submissions, contact info, activity)
- Search and filter across all entities
- Assign/reassign candidates and employers to other Operators
- Export data to CSV
- View own commission/payment info in Reports

## 3. Needs Action Section

### 3.1 Purpose
Display all items requiring immediate or upcoming attention, allowing Operators to prioritize their work efficiently.

### 3.2 Dropdown Filter

**Values:**
- **Today** (default) - Items due today
- **In next 7 days** - Items due within next week  
- **All** - All pending items

**Location:** Top-right of the section header

### 3.3 Criteria for "Needs Action"

Items appear in this list based on the following rules:

#### 3.3.1 New Candidates
- **Condition:** Status = "New" AND submitted within last 7 days
- **Why:** New candidates need initial contact within 7 days
- **Date shown:** Submission date

#### 3.3.2 Follow-up Due
- **Condition:** "Next action date" = today (or within selected dropdown range)
- **Why:** Candidates with scheduled follow-up actions
- **Date shown:** Next action date

#### 3.3.3 Unassigned Candidates
- **Condition:** No operator assigned
- **Why:** Work needs to be distributed
- **Date shown:** Submission date (action created same day)
- **Note:** These also appear in "New Candidates" if within 7 days

#### 3.3.4 Pending Employer Requests
- **Condition:** Status = "Pending"
- **Why:** Employer requests need review/processing
- **Date shown:** Request date

### 3.4 Table Structure

**Columns:**
- **Type** - Icon + label (Candidate/Employer)
- **Name/Company** - Person name or company name
- **Status** - Badge with color coding (New/Contacted/Screened/Pending)
- **Date** - Submission date or due date
- **Arrow (â†’)** - Navigation indicator

**Sorting:** Default sort by date (oldest first for urgency)

**Pagination:** Show 20 items, paginate if more

### 3.5 Interaction Behavior

**Row Click:**
- **Candidate row** â†’ Navigate to Candidate Public Profile page
- **Employer row** â†’ Navigate to Employer Public Profile page

**No bulk actions** (per requirements)

**Section Header:**
- Title is NOT clickable
- List displays inline on the landing page
- No modal/overlay

### 3.6 Mobile Responsive Behavior
- Table switches to card layout on small screens
- Each card shows: Type icon, Name, Status badge, Date, Arrow
- Cards stack vertically
- Swipe left/right for pagination

## 10. Technical Requirements

### 10.1 Responsive Design
- **Mobile-first approach**
- Needs Action table converts to cards on mobile
- Tiles stack vertically on small screens
- Filters collapse into expandable section on mobile
- Touch-friendly buttons and links (minimum 44px tap targets)

### 10.2 Performance
- Landing page loads in <2 seconds
- Tables paginate at 50 rows
- Search responds in <500ms (AJAX)
- Filters apply in <500ms
- Use database indexes on frequently queried fields

### 10.3 Security
- **Role verification:** Server-side check that user has Operator role
- **Capability checks:** Use WordPress capabilities for each action
- **CSRF protection:** WordPress nonces on all forms
- **SQL injection prevention:** Use $wpdb->prepare() for all queries
- **XSS prevention:** Escape all output with esc_html(), esc_url(), etc.
- **Audit logging:** Log all status changes, assignments, note additions

### 10.4 Browser Support
- Modern browsers (Chrome, Firefox, Safari, Edge - latest 2 versions)
- Graceful degradation for older browsers
- No IE11 support required

### 10.5 Accessibility
- WCAG 2.1 Level AA compliance
- Keyboard navigation support
- Screen reader friendly
- Proper ARIA labels
- Color contrast ratio 4.5:1 minimum

## 11. User Stories

### US-OP-1: View Action Items
**As an** Operator  
**I want to** see all items needing my attention when I log in  
**So that** I can prioritize my work efficiently

**Acceptance Criteria:**
- "Needs Action" section displays on landing page
- Shows correct count of items
- Items are categorized by type (Candidate/Employer)
- Default filter is "Today"
- List updates when dropdown filter changes

---

### US-OP-2: Filter Action Items by Timeframe
**As an** Operator  
**I want to** filter "Needs Action" by timeframe (Today/7 days/All)  
**So that** I can plan ahead and see what's coming up

**Acceptance Criteria:**
- Dropdown with three options: Today, In next 7 days, All
- Selecting an option updates the list without page reload
- Count updates to reflect filtered items
- Filter state persists during session

---

### US-OP-3: Quick Navigate to Profiles
**As an** Operator  
**I want to** click on a candidate/employer in "Needs Action" and go directly to their profile  
**So that** I can quickly take action

**Acceptance Criteria:**

- Clicking candidate row navigates to candidate detail page
- Clicking employer row navigates to employer detail page
- Navigation preserves context for back button

---

### US-OP-4: Search and Filter Candidates

**So that** I can find specific candidates quickly

**Acceptance Criteria:**
- Search box searches across name, email, LinkedIn, skills

- Applying filters updates table without page reload
- Reset button clears all filters
- Filter state persists in URL for bookmarking

### US-OP-5: Assign/Reassign Candidates
**As an** Operator  
**I want to** assign or reassign candidates to myself or other Operators  
**So that** work is distributed properly

**I want to** add internal notes to candidate profiles

## 13. Implementation Notes

### 13.1 Technology Stack
- **Page Builder:** Elementor (for layout and tiles)
- **Custom Widgets:** Custom Elementor widgets for tables and forms
- **Backend:** WordPress with custom capabilities
- **AJAX:** For search/filter without page reload
- **Database:** WordPress custom tables or post types

### 13.2 Data Model

**Consider creating:**
- `td_candidate` custom post type or table
- `td_employer` custom post type or table
- `td_scout` user meta or custom table
- `td_assignment` custom table (operator_id, candidate_id, assigned_date)
- `td_activity_log` custom table (entity_type, entity_id, action, actor, timestamp)
- `td_notes` custom table (entity_type, entity_id, author_id, note_text, timestamp)

### 13.3 WordPress Capabilities

**Create custom capability:** `manage_operator_portal`

**Assign to Operator role:**
- `manage_operator_portal`
- `read_candidates`
- `edit_candidates`
- `assign_candidates`
- `view_scouts`
- `export_data`

### 13.4 URL Structure

Preserve filter state in URLs:
- `/operators/candidates/?status=new&assigned=unassigned&date=7days&search=react`
- Allows bookmarking filtered views
- Enables back button to return to filtered state

### 13.5 AJAX Implementation

**Search/Filter AJAX:**
```javascript
jQuery(document).ready(function($) {
    $('#apply-filters').on('click', function() {
        var data = {
            action: 'filter_candidates',
            nonce: ajax_object.nonce,
            status: $('#status-filter').val(),
            assigned: $('#assigned-filter').val(),
            // ... other filters
        };
        
        $.post(ajax_object.ajax_url, data, function(response) {
            $('#results-table').html(response);
        });
    });
});
```
