## Overview

The Manager Actions page provides a complete 5-tab approval workflow for processing user registration requests across all user types (Candidate, Employer, Scout, Operator, Manager).

**Page URL:** `/managers/actions` (Page ID: 84)  
**Access:** Manager role only  
**Implementation:** Gutenberg page + custom mu-plugin shortcodes

---

## Tab Structure

### 1. New Tab (Blue Background)
**Status Filter:** Unassigned requests (`status='new'` OR `status='pending' AND assigned_to IS NULL`)

**Features:**
- View all new unassigned registration requests
- Shows: Request ID, Name, Role, Email, Phone, Type, Submitted date
- Action: **Assign to Me** button (➜ navy circle)

**Workflow:**
1. Manager sees list of new requests
2. Clicks "Assign to Me" button
3. Modal popup confirms assignment
4. Request moves to "Assigned" tab
5. Status changes to `pending` with `assigned_to` set to current manager

### 2. Assigned Tab (Yellow Background)
**Status Filter:** Assigned requests (`status='pending' AND assigned_to IS NOT NULL`)

**Features:**
- View requests assigned to any manager
- Shows: All columns + "Assigned To" column
- Highlights "Me" for current user's assignments
- Actions:
  - **Approve** button (✓ green circle) - Only if assigned to me
  - **Reject** button (✗ red circle) - Only if assigned to me
  - **Reassign to Me** button (➜ navy circle) - If assigned to someone else

**Workflow:**
- **For my assignments:**
  1. Click Approve → Creates user account, generates record_id, sends email
  2. Click Reject → Marks as rejected, sends rejection email
  3. Request disappears from tab (moves to Approved/Rejected)

- **For others' assignments:**
  1. Click Reassign to Me → Takes ownership
  2. Request now shows as "Me" in Assigned To column

### 3. Approved Tab (Green Background)
**Status Filter:** Approved requests (`status='approved'`, last 90 days)

**Features:**
- View all approved requests (90-day history)
- Shows: All columns including who approved and when
- Action: **Undo Approval** button (↶ orange circle)

**Workflow:**
1. Manager sees approved requests
2. Click Undo Approval → Reverts to `pending` status with `assigned_to` set
3. Request moves back to "Assigned" tab for reconsideration
4. Allows correction before user account activation

### 4. Rejected Tab (Red Background)
**Status Filter:** Rejected requests (`status='rejected'`, last 90 days)

**Features:**
- View all rejected requests (90-day history)
- Shows: All columns including who rejected and when
- Action: **Undo Rejection** button (↶ orange circle)

**Workflow:**
1. Manager sees rejected requests
2. Click Undo Rejection → Reverts to `pending` status with `assigned_to` set
3. Request moves back to "Assigned" tab for reconsideration
4. Allows giving requester another chance without new submission

### 5. All Tab (Gray Background)
**Status Filter:** All requests (last 90 days)

**Features:**
- Combined view of all statuses
- Shows appropriate action buttons based on current status
- Useful for overview and searching across all statuses

## Technical Implementation

### Database Table
**Table:** `td_user_data_change_requests`

**Key Fields:**
- `id` - Request ID (auto-increment)
- `status` - new, pending, approved, rejected
- `assigned_to` - User ID of assignee
- `assigned_by` - User ID who assigned
- `requested_role` - candidate, employer, scout, operator, manager
- `request_type` - register, update
- `record_id` - Generated on first approval (PENG-016)
- `submitted_date`, `updated_date` - Timestamps

### MU-Plugin
**File:** `wp-content/mu-plugins/manager-actions-display.php`  
**Version:** 1.0.0

**Shortcode:** `[manager_actions_table]`

**Parameters:**
- `status` - new, assigned, approved, rejected, all (required)
- `limit` - Max results (default: 50)
- `days` - History range for approved/rejected (default: 90)

**Example Usage:**
```html
[manager_actions_table status="new" limit="50"]
[manager_actions_table status="assigned" limit="50"]
[manager_actions_table status="approved" limit="50" days="90"]
[manager_actions_table status="rejected" limit="50" days="90"]
[manager_actions_table status="all" limit="100" days="90"]
```

### AJAX Handlers (Reused from user-requests-display.php)
All AJAX endpoints already implemented:
- `td_assign_request` - Assign request to user
- `td_approve_request` - Approve request, generate record_id
- `td_reject_request` - Reject request
- `td_undo_approve` - Undo approval
- `td_undo_reject` - Undo rejection

### JavaScript Behavior
- **No blocking alerts** - All actions execute immediately
- **Non-blocking notifications** - Success/error messages via tdShowNotification
- **Modal for assignment** - "Assign to Me" shows popup confirmation
- **Row removal** - Approved/rejected rows fade out after action
- **Cache busting** - All event handlers use `.off().on()` pattern

## Security

### Access Control
- **Role Check:** Manager or Administrator only
- **Nonce Verification:** All AJAX calls require valid nonce
- **Assignment Validation:** Can only assign to Manager/Operator roles
- **Audit Logging:** All actions logged to `td_audit_log` table

### Data Validation
- Request ID validation (integer check)
- User role validation before AJAX execution
- Status validation (can only undo from approved/rejected)
- Assignment ownership checks (approve/reject only if assigned to me)

## User Experience

### Assignment Flow
1. Manager lands on "New" tab
2. Sees unassigned requests in table
3. Clicks navy arrow (➜) button
4. Modal appears: "Assign to Me" / "Cancel"
5. Clicks "Assign to Me"
6. Modal closes, success notification appears
7. Page reloads, request now in "Assigned" tab

### Approval Flow
1. Manager switches to "Assigned" tab
2. Sees only their assignments
3. Reviews request details in row
4. Clicks green checkmark (✓) button
5. Button disables, opacity reduces
6. Success notification appears
7. Row fades out and removes from table
8. User account created in background

### Rejection Flow
1. Manager on "Assigned" tab
2. Clicks red X (✗) button
3. Button disables immediately
4. Success notification appears
5. Row fades out and removes
6. No reason required (simplified UX)

### Undo Flow
1. Manager realizes mistake
2. Switches to "Approved" or "Rejected" tab
3. Finds the request (90-day history)
4. Clicks orange undo (↶) button
5. Request returns to "Assigned" tab
6. Can now take different action

## Troubleshooting

### Common Issues

**Problem:** Buttons still showing as pill-shaped  
**Solution:** Hard refresh browser (Ctrl+F5), flush WordPress cache, check CSS !important flags

**Problem:** Blocking alerts still appearing  
**Solution:** Clear browser cache, verify mu-plugin updated, check JavaScript console for errors

**Problem:** Empty table shows no requests  
**Solution:** Check if `td_user_data_change_requests` table exists, verify Forminator sync working

**Problem:** Assign modal not appearing  
**Solution:** Check jQuery loaded, verify tdShowNotification function exists, check console errors

**Problem:** Actions not executing  
**Solution:** Verify nonce valid, check AJAX endpoints registered, review network tab in DevTools
